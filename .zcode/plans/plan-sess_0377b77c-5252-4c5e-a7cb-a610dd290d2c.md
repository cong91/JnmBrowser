# VI-OTP SMS/OTP Integration Plan

> **Goal:** Implement a general SMS/OTP service abstraction with a concrete VI-OTP provider. Support both standalone dialog usage AND auto-registration engine integration.

## User Requirements Confirmed
- **Use case:** Both standalone dialog + auto-registration integration
- **API token:** Stored in App Settings (encrypted `.dat` pattern)
- **Extensibility:** Must be general so other SMS providers can be plugged in later

---

## Architecture: Mirroring `EmailService` Pattern

```
src-tauri/src/sms/
├── mod.rs          # SmsService trait + shared types (Network, Service, NumberInfo, OtpInfo, etc.)
├── error.rs        # SmsServiceError enum
├── viotp.rs        # ViotpService (VI-OTP API concrete implementation)
└── commands.rs     # Tauri commands for standalone dialog usage
```

### SmsService Trait (sync, like EmailService)

| Method | Description |
|--------|-------------|
| `get_balance()` → `f64` | Check account balance |
| `get_networks(country?)` → `Vec<Network>` | List carriers: MOBIFONE, VIETTEL, etc. |
| `get_services(country?)` → `Vec<Service>` | List services: Facebook, Momo, Shopee, etc. |
| `request_number(service_id, network?, prefix?, except_prefix?, country?)` → `NumberInfo` | Rent a phone number |
| `get_otp(request_id, timeout_secs)` → `OtpInfo` | Poll for OTP code (with deadline) |
| `get_history(service?, status?, limit?, from?, to?)` → `Vec<HistoryEntry>` | Rental history |
| `check_health()` → `bool` | API reachability |

### ViotpService — Concrete Implementation
- Base URL: `https://api.viotp.com`
- API token passed via query param `?token=...`
- Uses `reqwest` with `block_on` helper (same pattern as GmailCdkService)
- All API calls are GET requests
- Status code handling: 200=OK, 401=auth error, 429=limit, -1=error, -2=insufficient balance, -3=out of numbers, -4=service unavailable

## Implementation Steps (10 tasks)

### Step 1: Create `sms` module with trait + types
**Files:** `src-tauri/src/sms/mod.rs`, `src-tauri/src/sms/error.rs`
- Define `SmsService` trait (sync, Send + Sync)
- Define shared types: `Network`, `Service`, `NumberInfo`, `OtpInfo`, `HistoryEntry`, `NumberRequest`, `HistoryQuery`
- Define `SmsServiceError` enum (mirrors EmailServiceError)

### Step 2: Implement `ViotpService`
**File:** `src-tauri/src/sms/viotp.rs`
- `ViotpService` struct with `api_token: String`
- Implement all `SmsService` methods against VI-OTP API
- Use `reqwest` + `block_on` helper (same as GmailCdkService)
- Polling loop for `get_otp` with configurable interval (3s default) and deadline

### Step 3: Register module in lib.rs + Cargo.toml
**Files:** `src-tauri/src/lib.rs`, `src-tauri/Cargo.toml`
- Add `pub mod sms;` to `lib.rs`
- Verify reqwest already in dependencies (it is)

### Step 4: Add SMS API token to settings (encrypted storage)
**Files:** `src-tauri/src/settings_manager.rs`
- Add `sms_api_token: Option<String>` field to `AppSettings` (the "display" field for UI)
- Add encrypted store/load/remove functions for `sms_token.dat` with magic header `b"DBVIO"`
- Add Tauri commands: `get_sms_api_token`, `set_sms_api_token`, `remove_sms_api_token`
- Follow existing `api_token` / `mcp_token` / `sync_token` pattern exactly

### Step 5: Create Tauri commands for standalone SMS usage
**File:** `src-tauri/src/sms/commands.rs`
- `sms_get_balance(token)` → balance
- `sms_get_networks(token, country?)` → networks list
- `sms_get_services(token, country?)` → services list
- `sms_request_number(token, service_id, network?, prefix?, except_prefix?, country?)` → number info
- `sms_get_otp(token, request_id, timeout_secs?)` → OTP code
- `sms_get_history(token, service?, status?, limit?, from_date?, to_date?)` → history
- All commands follow `#[tauri::command]` async pattern with internal `spawn_blocking`
- Register in `lib.rs` generate_handler

### Step 6: Frontend types + locales
**Files:** `src/types.ts`, 7 locale files under `src/i18n/locales/`
- Add TypeScript interfaces: `SmsNetwork`, `SmsService`, `SmsNumberInfo`, `SmsOtpInfo`, `SmsHistoryEntry`
- Add i18n keys for SMS dialog: `sms.*` namespace
- Update all 7 locales (en, es, fr, ja, pt, ru, zh)

### Step 7: Standalone SMS verification dialog
**File:** `src/components/sms-verification-dialog.tsx` (new)
- Country selector (Vietnam / Laos)
- Balance display + refresh
- Network/carrier multi-select checkboxes
- Service dropdown (fetched from API)
- "Request Number" button → shows phone number + request_id
- OTP polling: auto-poll with countdown, manual refresh
- Display OTP code prominently with copy button
- History tab: table of past rentals with status

### Step 8: Add SMS config to auto-registration
**Files:** `src-tauri/src/auto_register/types.rs`, `src/hooks/use-registration-events.ts`
- Add `SmsConfig { sms_service: Option<String>, sms_token: Option<String>, sms_service_id: Option<u32>, sms_network: Option<String>, sms_country: Option<String> }` to `RegistrationConfig`
- Add SMS-related `RegistrationStep` variants: `RequestingSmsOtp`, `PollingSmsOtp`, `VerifyingSmsOtp`
- Update `RegistrationResult` with optional `phone_number` and `sms_otp` fields
- Mirror in frontend TypeScript types

### Step 9: Integrate SMS into auto-registration engine
**Files:** `src-tauri/src/auto_register/engine.rs`, `src-tauri/src/auto_register/commands.rs`
- Thread `Option<&dyn SmsService>` through `run()` → `run_once()` → browser flow
- Handle `PageType::AddPhone` in state machine:
  1. Call `sms_service.request_number()` to get a phone number
  2. Submit phone number to ChatGPT "add phone" form via CDP
  3. Call `sms_service.get_otp()` to poll for SMS code
  4. Submit OTP code to ChatGPT
- Pass ViotpService instance from `commands.rs` when SMS config is present

### Step 10: Verify, format, lint, test

- `pnpm lint` — Biome + tsc + clippy + typos
- `pnpm test` — Rust unit + sync E2E

## Key Design Decisions
- **Trait methods are sync** (like EmailService) — engine calls inside spawn_blocking; standalone Tauri commands wrap in spawn_blocking
- **Token is stored encrypted** via Argon2 + AES-256-GCM (same as api_token/mcp_token/sync_token)
- **VI-OTP base URL hardcoded** as `https://api.viotp.com` — future providers can have their own URLs
- **Polling interval: 3s** (same as GmailCdkService), configurable timeout
- **Auto-registration engine** takes `Option<&dyn SmsService>` — SMS is optional, email flow unchanged when None
