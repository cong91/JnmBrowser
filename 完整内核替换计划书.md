## 1A. 内核下载 Manifest 设计

本次调整：manifest 结构保持和 Wayfern 官方 `wayfern.json` 一致，避免下载器和版本管理做额外适配。

### 本地模拟文件

```text
/Users/lijinmu/Documents/New project/donutbrowser/fingerprint-chromium.json
```

该文件用于模拟：

```text
https://donutbrowser.com/wayfern.json
```

当前正式 manifest 地址：

```text
https://raw.githubusercontent.com/JnmHub/JnmBrowser-chromium/main/fingerprint-chromium.json
```

已验证该地址当前可访问，JSON schema 与 Wayfern manifest 一致。

### Manifest 结构

保持最小结构：

```json
{
  "version": "142.0.7444.175",
  "downloads": {
    "linux-x64": "https://github.com/adryfish/fingerprint-chromium/releases/download/142.0.7444.175/ungoogled-chromium-142.0.7444.175-1-x86_64_linux.tar.xz",
    "linux-arm64": null,
    "macos-x64": "https://github.com/adryfish/fingerprint-chromium/releases/download/142.0.7444.175/ungoogled-chromium_142.0.7444.175-1.1_macos.dmg",
    "macos-arm64": "https://github.com/adryfish/fingerprint-chromium/releases/download/142.0.7444.175/ungoogled-chromium_142.0.7444.175-1.1_macos.dmg",
    "windows-x64": "https://github.com/adryfish/fingerprint-chromium/releases/download/142.0.7444.175/ungoogled-chromium_142.0.7444.175-1.1_windows_x64.zip",
    "windows-arm64": null
  }
}
```

### 为什么不扩展字段

之前多加了 `name/channel/license/archive_type/executable_relative_path/launch_defaults`，虽然信息更完整，但会导致和 Wayfern manifest 不一致。

现在改成最小结构，优点是：

- 复用现有 Wayfern JSON parser 更简单。
- 未来网络地址替换成本更低。
- 下载器只关心 `version + downloads[platform_key]`。
- 解包类型可以从 URL 后缀判断：`.dmg` / `.zip` / `.tar.xz`。
- executable 路径由安装后扫描决定，不写死在 manifest。

### 代码读取策略

新增配置优先级：

```text
1. 环境变量 JNM_FINGERPRINT_CHROMIUM_MANIFEST_URL
2. GitHub raw manifest：https://raw.githubusercontent.com/JnmHub/JnmBrowser-chromium/main/fingerprint-chromium.json
3. 本地开发 fallback 文件 fingerprint-chromium.json
```

第一阶段默认读取 GitHub raw manifest：

```text
https://raw.githubusercontent.com/JnmHub/JnmBrowser-chromium/main/fingerprint-chromium.json
```

本地文件只作为开发 fallback：

```text
/Users/lijinmu/Documents/New project/donutbrowser/fingerprint-chromium.json
```

### Manifest 验收要求

- [ ] JSON schema 和 Wayfern manifest 保持一致。
- [ ] JSON 解析失败时返回明确错误。
- [ ] 当前平台 entry 为 `null` 时返回“不支持当前平台”。
- [ ] 当前平台 entry 缺失时返回明确 platform key。
- [ ] 本地 manifest 和网络 manifest 使用同一套 parser。
- [ ] 后期网络请求失败时可以 fallback 到最近一次缓存。

### 当前跨平台结论（2026-05-02 更新）

- macOS：`macos-arm64` / `macos-x64` 已接通 manifest、下载、DMG 解包链路；其中本机已真实跑通 `macos-arm64`
- Linux：`linux-x64` 已有 manifest 下载项，下载文件名 / 解包格式 / executable 探测路径已补单测覆盖
- Windows：`windows-x64` 已有 manifest 下载项，下载文件名 / ZIP 解包 / executable 探测路径已补单测覆盖
- 暂不支持：`linux-arm64` / `windows-arm64`，因为当前 manifest 仍为 `null`

这意味着当前阶段可按：

- **macOS**
- **Linux x64**
- **Windows x64**

来做“多系统兼容”收口；如果后面还要 **Linux ARM64 / Windows ARM64**，那是后续新增平台支持，不是本轮主线里漏接。

## 2. 替换策略

### 选择：直接替换，不新增浏览器类型

本次不新增 `fingerprint-chromium` profile 类型，避免前端、数据库、profile schema 大规模迁移。

采用策略：

```text
原来选择 wayfern 的地方
        ↓
实际启动 fingerprint-chromium 142
        ↓
继续使用 Chromium CDP 自动化链路
```

也就是说：

- UI / API / MCP 层可以继续把它当作 `wayfern` 或 Chromium 自动化内核使用。
- 下载目录、查找目录、启动参数改为 fingerprint-chromium。
- Wayfern token、Wayfern terms、Wayfern paid gate 相关逻辑逐步从自动化主链路移除。
- 由于目标内核版本是 Chromium 142，而旧 Wayfern 可能是 Chromium 146，第一阶段不要直接复用旧 Wayfern user-data-dir，避免 Chromium profile 降级损坏。
- 用户确认旧 Wayfern 数据只是本机测试数据，不需要迁移；因此第一阶段采用“新内核新目录，旧目录保留不用”的最简策略。

### 为什么这样做

优点：

- 最小化 profile 兼容成本。
- MCP 工具不用大改。
- 现有 Chromium CDP 逻辑可复用。
- `Runtime.evaluate` 恢复可用。
- 保留 JnmBrowser 自己的 profile、代理、cookie、storage、MCP 自动化体系。

代价：

- 代码里 `wayfern` 这个名字短期内会和实际内核不完全一致。
- 后续最好再做一次命名清理，把内部名改为 `fingerprint_chromium` 或 `chromium_fingerprint`。

### 3.4 指纹配置映射

文件可能涉及：

### 映射表

| JnmBrowser 配置 | fingerprint-chromium 参数 | 优先级 |
|---|---|---:|
| fingerprint seed / profile id hash | `--fingerprint=<u32>` | P0 |
| OS 类型 | `--fingerprint-platform=macos/windows/linux` | P0 |
| OS 版本 | `--fingerprint-platform-version=<version>` | P1 |
| 浏览器品牌 | `--fingerprint-brand=<brand>` | P1 |
| 浏览器版本 | `--fingerprint-brand-version=<version>` | P1 |
| CPU 核心数 | `--fingerprint-hardware-concurrency=<n>` | P0 |
| 语言 | `--lang=<lang>` | P0 |
| Accept-Language | `--accept-lang=<langs>` | P0 |
| 时区 | `--timezone=<tz>` | P0 |
| WebRTC 防泄漏 | `--disable-non-proxied-udp` | P0 |
| GPU vendor | `--fingerprint-gpu-vendor=<vendor>` | P2 |
| GPU renderer | `--fingerprint-gpu-renderer=<renderer>` | P2 |
| 禁用特定 spoofing | `--disable-spoofing=font,audio,canvas,clientrects,gpu` | P2 |

### seed 生成规则

如果 profile 已有稳定 seed，直接使用。

如果没有，建议用 profile id 稳定生成：

```text
seed = first_4_bytes(sha256(profile_id)) as u32
```

要求：

- 同一个 profile 每次启动 seed 一致。
- 不同 profile seed 尽量不同。
- 不要每次启动随机，否则 fingerprint 不稳定。

验收：

- 同一 profile 多次启动，CreepJS / BrowserLeaks 关键指纹保持稳定。
- 修改 profile fingerprint 后，启动参数同步变化。

```json

JNM_USE_FINGERPRINT_CHROMIUM=0
```

恢复旧 Wayfern resolver。

## 6. 风险清单

| 风险 | 影响 | 处理 |
|---|---|---|
| DMG 内 app 名称不是 Chromium.app | 下载后找不到 executable | 解包后扫描 `*.app` |
| executable 名称不同 | 启动失败 | 扫描 `Contents/MacOS` 下可执行文件 |
| Cookie DB schema 差异 | cookie 导入导出失败 | 实测后兼容 schema |
| profile 目录与 Wayfern 有差异 | 登录态/配置不兼容 | 新 profile 路径隔离测试 |
| 指纹参数部分无效 | 指纹不符合预期 | CreepJS/BrowserLeaks 验证 |
| 代理认证不支持 | 无法直连账号密码代理 | 继续走 donut-proxy |
| macOS Gatekeeper/quarantine | app 无法启动 | 下载后执行 `xattr -dr com.apple.quarantine` |
| 名称仍叫 wayfern | 后续维护混乱 | 第二阶段重命名内部 engine |
| Chromium 146 profile 被 142 打开 | profile 损坏/无法回滚 | fingerprint-chromium 使用全新独立 user-data-dir，第一阶段不做旧数据迁移 |
| macOS dmg 实际架构不匹配 | Apple Silicon 启动失败或依赖 Rosetta | Phase 0 用 `file`/`lipo` 验证，失败时禁止标记已安装 |
| 远程 manifest 不可用 | 无法下载/升级内核 | 使用缓存 manifest 和本地 fallback |
| 半安装目录被误识别 | 后续启动失败 | 下载到临时目录，验证 executable 后原子移动到正式目录 |
| 进程名变化 | kill/status 失效 | 根据 `--user-data-dir` 和 executable path 匹配 |
| 扩展加载参数丢失 | 扩展能力回归 | 保留并测试 `--load-extension` / extension group |
